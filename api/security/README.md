# Security API

HTTP handlers for `/v1/security` requests plus scaffolding for a future vulnerability analysis engine. The current implementation returns deterministic mock data so the surrounding controller, CLI, and tests can exercise the contracts without relying on external scanners.

## Key Takeaways
- Security routes are thin Fiber handlers backed by a `SecurityEngine` stub; there is no live integration with Grype, NVD, or Mods execution yet.
- The package focuses on struct definitions and data shaping so downstream systems can develop against stable JSON schemas.
- Configuration helpers only expose NVD toggles (`NVD_ENABLED`, `NVD_API_KEY`, etc.); they are parsed but not consumed by runtime logic.
- Example recipes under `examples/` demonstrate the expected payload shapes for future automation but are not wired into the handlers.

## HTTP Endpoints
- `POST /v1/security/scan` – Parses a `target` and `scan_type`, returns a canned security report with vulnerability, risk, and summary blocks.
- `POST /v1/security/mods/plan` – Accepts a list of vulnerability identifiers and returns a fabricated remediation plan (steps, effort, timeline).
- `GET /v1/security/report` and `GET /v1/security/report/:id` – Emit the same mock report structure used by the scan endpoint, keyed by the supplied identifier.
- `GET /v1/security/compliance` – Produces a placeholder compliance snapshot with OWASP/NIST/CIS scores.

All responses are generated by the handler itself so they remain stable for tests. Unsupported paths such as sandbox management or SBOM routes are intentionally absent—SBOM lives in its own package, and sandbox lifecycle belongs to Mods.

## Implementation Notes
- `Handler` wires the routes and wraps a `SecurityEngine` instance. The engine exposes methods for scanning and remediation planning but currently fabricates data in-memory.
- `security_engine.go` defines helpers for prioritising vulnerabilities, building remediation timelines, and estimating effort. The calculations only depend on the incoming slice and do not call external services.
- `security_engine_types.go` and `common_types.go` host the JSON DTOs shared by planned subsystems (CVE metadata, mod recipes, SBOM analysis, metrics, etc.).
- `config.go` loads NVD settings from the environment, supporting future wiring of a real CVE database client.
- `pattern_matcher.go` and related interfaces are placeholders that always return synthetic patterns.

## Files & Folders
- `handler.go` / `handler_security.go` – Fiber setup plus the mock HTTP responses.
- `security_engine.go` – Stubbed engine logic and helper methods (`prioritizeVulnerabilities`, `createModTimeline`, `calculateEffort`).
- `security_engine_types.go` / `common_types.go` / `shared_types.go` – Data structures used by the handlers and tests.
- `config.go` / `config_validator_test.go` – Configuration loading helpers and coverage for env overrides.
- `examples/` – Sample YAML recipes for future automated remediation workflows.
- `security_engine_test.go` – Ensures the stub implementations preserve expected shapes and basic validations.

## Testing
- `go test ./api/security/...` executes the engine unit tests and config validation. There are no integration tests because the handlers do not reach external services yet.
- The mock responses inside `handler_security.go` are intentionally stable so higher-level API tests can assert on fields without requiring fixtures.

## Limitations & Next Steps
- No real vulnerability scanning or SBOM correlation is performed; `SecurityEngine` must be extended with concrete `CVEDatabase`, `RiskAnalyzer`, and `SBOMSecurityAnalyzer` implementations.
- Mods integration (submitting remediation runs, surfacing logs/artifacts) happens exclusively through the Mods API—this package only returns descriptive data.
- Sandbox lifecycle management remains a future concern and is not exposed under `/v1/security`.
