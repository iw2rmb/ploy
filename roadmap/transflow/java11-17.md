## Java 11 → 17 Migration (Key Takeaways)

Scenario (Controller‑centric)
- User runs ploy CLI locally with only `PLOY_CONTROLLER` configured.
- All work (repo clone, planning, HCL rendering, Nomad job submission, artifact storage) executes on the VPS by the Controller.
- Steps executed on the VPS:
  1) Apply initial OpenRewrite step (`org.openrewrite.java.migrate.Java11toJava17`) and run build gate (no deploy).
  2) On build failure, the Controller triggers the LangGraph planner job (inputs: last_error, repo metadata, KB snapshot).
  3) Planner emits parallel healing options; the Controller orchestrates fanout; first‑success‑wins; reducer finalizes.
  4) Controller returns status + artifact URLs (plan.json, next.json, diff.patch) to the local CLI.

Parallel Healing Options (job types)
- human-step: pause; wait for human MR/commit; then build-check.
- llm-exec: generate a minimal diff to fix compile (e.g., imports, API usage) → apply → build-check.
- orw-gen → openrewrite: generate/resolve an ORW recipe (class/coords) that targets the failure → apply → build-check.

Tools & Context
- Web search (public) and corporate wiki (internal) for error patterns and version guidance.
- KB retrieval: similar error signatures → suggested ORW recipes or patch fingerprints.
- Local search (ripgrep) over repo; dependency manifest parsing (pom.xml/gradle).

Dependency Issues (common during JDK upgrades)
- Symptoms: incompatible transitive versions; APIs moved/removed; library must be bumped for JDK 17.
- Default strategies supported:
  1) Search: locate known-good versions for the stack or conflicting libs.
  2) Bump: try controlled version bumps until error disappears.

Small Verification App (recommended)
- Create a minimal Java app with dependencies known to fail under JDK 17 (e.g., old Mockito/JUnit, JAXB usage).
- Exercise: run ORW recipe → build gate (Lane C) → induce failure → observe planner branches → verify a branch fixes build; optional deploy can be added later.

Does current design handle dependency issues?
- Yes, at a basic level:
  - llm-exec can edit `pom.xml`/Gradle to bump versions or add replacements; validated by build gate.
  - orw-gen can propose migration recipes that rewrite usage to supported APIs.
  - KB can promote known version pairs (JDK 17 ↔ library X@vY) for instant suggestions.
- Caveats:
  - Blind bumping can introduce subtle runtime issues; keep diffs minimal and run compile-only gate (MVP).
  - Multiple interdependent bumps may require iterative planning; planner supports bounded retries.

Targeted Enhancements (optional, high value)
- dep-bump branch type:
  - Use Maven Versions Plugin/Gradle Version Catalog to suggest or test bumps; write minimal diff; build-check.
  - Guardrails: allowlist groupIds/artifacts; max bump range; prefer minor/patch upgrades first.
- dep-graph analyzer:
  - Parse `mvn dependency:tree` / Gradle dependencies; detect conflicts; consult KB for resolved sets.
- KB curation for JDK 17:
  - Seed with common lib mappings (e.g., JAXB/Jakarta transitions; JUnit4→5; Mockito versions compatible with JDK 17).

Flow (end‑to‑end, Controller‑centric)
1) Local: `ploy mod run -f mod.yaml` sends a request to the Controller API (PLOY_CONTROLLER) and streams status.
2) Controller clones the repo to VPS, applies ORW Java11→17; runs build gate. If build fails, it submits planner job via the Nomad Job Manager wrapper.
3) Controller receives plan.json, selects branches; submits `llm-exec`/`orw-gen`/`human-step` via wrapper; first success wins; cancels others.
4) Controller persists artifacts (plan.json, next.json, diff.patch) to SeaweedFS and returns URLs + summary to the client.

Safety & Policy
- Diff-only edits; path allowlists; timeout/budget bounds per branch.
- Prefer minimal changes; record outcomes for continuous improvement (KB).

## Verification Checklist

- Local config parses: `go run ./cmd/ploy mod run -f test-mod-java11to17.yaml --dry-run -v`
- Planner assets render: `go run ./cmd/ploy mod run -f test-mod-java11to17.yaml --render-planner -v`
- Plan HCL renders (no submit by default): `go run ./cmd/ploy mod run -f test-mod-java11to17.yaml --plan -v`
- Reducer HCL renders (optional): `go run ./cmd/ploy mod run -f test-mod-java11to17.yaml --reduce -v`
- Execution assets for first options render:
  - LLM branch: `go run ./cmd/ploy mod run -f test-mod-java11to17.yaml --plan --exec-llm-first -v`
  - ORW branch: `go run ./cmd/ploy mod run -f test-mod-java11to17.yaml --plan --exec-orw-first -v`
- Diff application gate (with diff artifact path/url provided):
  - `go run ./cmd/ploy mod run -f test-mod-java11to17.yaml --apply-first -v` with `MODS_DIFF_PATH` or `MODS_DIFF_URL` set

## Local Usage (Thin Client)

The local CLI acts as a thin client. Set PLOY_CONTROLLER and call:

```bash
# ensure PLOY_CONTROLLER=https://api.dev.ployman.app/v1, then run:
ploy mod run -f test-mod-java11to17.yaml -v
```

The Controller performs all VPS‑side work. The CLI displays progress and, on completion, prints artifact URLs and a summary. No local Nomad/Consul/SeaweedFS access is needed.

## Controller Responsibilities (VPS)

- Clone repositories, render HCL, and submit Nomad jobs using `/opt/hashicorp/bin/nomad-job-manager.sh` (never raw Nomad SDK/CLI on VPS).
- Store artifacts (plan.json, next.json, diff.patch) in SeaweedFS and return URLs to the client.
- Use the internal Docker Registry for all job images (langgraph-runner, openrewrite-jvm) via environment defaults.
- Expose concise status endpoints for client polling/streaming.

Validation targets:
- The local CLI only requires PLOY_CONTROLLER; no local access to Nomad/Consul/SeaweedFS is needed.
- MR created/updated on GitLab with branch `workflow/<id>/<timestamp>` (optional, per config).
- Build gate passes (compile only; no deploy); if initial build fails, healing plan produced; at least one branch succeeds; reducer outputs `{ "action": "stop" }`.
- diff.patch is non‑empty for the orw‑apply branch in this scenario (pass criteria).

Implementation delta to support thin client
- CLI (internal/mods):
  - Remove direct HCL rendering/submission and Nomad SDK usage in the client for production paths.
  - Add client requests to Controller: create transflow run, poll status, download artifacts.
  - Keep local `--dry-run` validation only (no submission).
- API/Controller (server):
  - Add endpoints: POST /v1/mods/run (start), GET /v1/mods/status/:id, GET /v1/mods/artifacts/:id (plan.json, next.json, diff.patch).
  - Move HCL rendering and Nomad job submission to server using the job manager wrapper.
  - Persist artifacts to SeaweedFS and return signed or direct URLs.
  - Configure default images via env (MODS_*_IMAGE) pointing to the internal Docker Registry.
- VPS policy enforcement:
  - Never use raw Nomad SDK/CLI in the server; use `/opt/hashicorp/bin/nomad-job-manager.sh` exclusively.
  - Ensure GitLab tokens and registry creds are available only on VPS.
